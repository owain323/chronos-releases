# TaskNest 模块评审清单

> **用途：** 每次开发或修改一个模块后，逐项检查。不是建议，是硬门槛。
> **时机：** `git commit` 之前，配合 pre-commit 自动检查（tsc + vitest + prettier）一起用。

---

## 检查项

### 1. 有没有重复代码？

- [ ] 同一个逻辑是否出现在两个以上文件中？
- [ ] 相同的 JSX 结构是否可以在组件层抽一个共享组件？
- [ ] 相同的 Tailwind class 组合是否可以用 `@apply` 或工具类抽象？

> 反例：`bg-sky-600 hover:bg-sky-700 text-white` 出现在 15 个文件里——这是按钮的 primary variant，应该由 Button 组件的 variant 控制，而不是每个地方手写。

### 2. 有没有破坏模块边界？

- [ ] 前端组件是否直接 import 了 `server/` 目录下的文件？
- [ ] `shared/` 里的类型定义是否被前端和后端都能用，且不依赖任何一方的实现？
- [ ] 页面组件 (`pages/`) 是否直接调用了数据库操作或文件系统？

> 边界规则：
>
> ```
> client/  ← 只能 import shared/ 和自身
> server/  ← 只能 import shared/ 和自身
> shared/  ← 不 import 任何一方
> ```

### 3. 有没有把业务逻辑写进组件？

- [ ] 组件是否只负责渲染和用户交互？
- [ ] 数据转换、计算、格式化是否在 `lib/` 或自定义 hook 里？
- [ ] API 调用是否通过 tRPC client（`@/lib/trpc`）而不是原生 fetch？

> 反例：在 `TaskList.tsx` 里直接写 `tasks.filter(...).map(...).reduce(...)` 做业务计算——应该抽出到 `@/lib/taskUtils.ts`。

### 4. 有没有新增技术债？

- [ ] `as any` 是否只在有 Zod 运行时校验的地方出现？
- [ ] `// TODO` 是否附带了日期和责任人？
- [ ] `console.log` 是否已替换为 `toast` 或结构化日志？
- [ ] 新依赖是否测试过 `npm run dev` 能正常启动？
- [ ] 新数据库列是否加了对应的 TypeScript 类型声明？

> 技术债不是不能有，是必须有记录。不要留"以后再说"但不写下来的东西。

---

## 快速使用

开发完一个功能后，在代码 diff 旁边过一遍这 4 个问题。如果可以诚实地说"没有"，就可以提交。

如果发现有一个"有"，先在当前 PR/branch 里修掉，不要推到下一个迭代。

---

## 与自动化检查的关系

| 检查手段              | 负责什么                                       |
| --------------------- | ---------------------------------------------- |
| **Husky pre-commit**  | tsc 零错误、vitest 全绿、prettier 格式统一     |
| **GitHub Actions CI** | 远程二次验证，防本地环境差异                   |
| **Review Checklist**  | 人肉检查：重复代码、模块边界、业务逻辑、技术债 |

三者不互相替代。**自动化拦语法，人眼拦设计。**
