# TaskNest 项目协作反馈

## 背景

以下是我在接手时发现并修复的 6 个问题。这些问题的共同特点是：**方向是对的，但执行时忽略了项目已有的约束和上下文**。整理出来供参考。

---

## 问题 1：数据库层被整体回退

**现象：** `server/db.ts` 从 `better-sqlite3`（SQLite）被重写为 `drizzle-orm/mysql2`（MySQL）。项目没有 MySQL 服务，所有 API 请求报 `ECONNREFUSED`，整个网站无法使用。

**根因：** 改动前没有确认项目当前的数据库架构。TaskNest 设计为**本地零配置运行**，SQLite 是刻意选择——不需要安装数据库服务，数据库就是项目根目录下的一个文件 `tasknest.db`。

**建议：**

- 接手一个项目时，先看 `drizzle.config.ts` 的 `dialect` 字段、看 `.env` 中的 `DATABASE_URL` 格式、看 `package.json` 中的数据库驱动依赖。这是确认数据库方案最快的方式。
- 除非有明确的技术决策变更，否则不要替换数据库层。
- 如果要新增功能，在已有的数据库方案上加，而不是换。

---

## 问题 2：search.ts 引用不存在的导出

**现象：** `server/routers/search.ts` 引入 `isMemoryDb()` 和 mock 数据数组，但这些新加的导出和函数名在实际的 db.ts 中不存在，导致启动时 `SyntaxError`，服务直接起不来。

**根因：** 修改了 db.ts 后，没有检查其他模块的 import 是否仍然有效。search.ts 依赖了 db.ts 中不存在的 API。

**建议：**

- 每改一个文件，搜一下项目里还有哪些文件 `import` 了它。IDE 的「查找引用」(Find References) 功能可以快速定位。
- 尤其要注意被多个模块引用的核心文件（如 `db.ts`），它的导出接口变动会级联到所有依赖方。

---

## 问题 3：AnalyticsPanel.tsx JSX 语法错误

**现象：** 第 165 行出现 `</div>"p-4 bg-muted rounded-lg">`——一个字符串字面量被放在 JSX 闭合标签之后，这违反了 JSX 语法规则，Vite 编译时报错。

**根因：** 大概率是编辑/复制时遗落的文本碎片。

**建议：**

- 修改 JSX 后，用 `npm run build` 或关注 Vite 开发服务器的终端输出，它能即时报告这类语法错误。
- 如果是复制粘贴，确保没有残留的文本节点。

---

## 问题 4：`xlsx` 包与 Vite 不兼容

**现象：** Dashboard.tsx 添加了 `import * as XLSX from "xlsx"`，但 Vite 报 `Failed to resolve entry for package "xlsx"`，因为 `xlsx@0.18.5` 的 `package.json` 中 `"module": "xlsx.mjs"` 指向一个不存在的文件，Vite 的 ESM 解析失败。

**根因：** `xlsx` 这个包年久失修，它的 ESM 入口声明与实际发布的内容不符。这是社区已知问题。

**建议：**

- 添加新依赖前，确认它能否在目标打包器（Vite/Webpack）下正常工作。检查方式：`import X from "xxx"` 在 Vite dev server 下能否正常 resolve。
- 对这种入口文件缺失的问题，最快解决方式是在 `vite.config.ts` 的 `resolve.alias` 中配一个别名，指向实际存在的文件。
- 或者换用社区维护更好的替代包（项目已有 `exceljs`，也支持 Excel 导出）。

---

## 问题 5：新增数据库表未创建

**现象：** `drizzle/schema.ts` 中新增了 `customers` 和 `customerContacts` 两张表的定义，查询代码也写了，但实际 SQLite 数据库文件中没有这两张表，查询时仍然会报错。

**根因：** Drizzle ORM 的 schema 定义只是 TypeScript 层面的类型声明，它不会自动在 SQLite 中创建物理表。需要手动执行建表语句或使用 drizzle-kit 做 migration。

**建议：**

- 在 SQLite 场景下，新增表需要在 `scripts/` 目录下的初始化脚本中添加 `CREATE TABLE IF NOT EXISTS` 语句。
- 如果项目有 migration 流程（drizzle-kit），用 `drizzle-kit push` 或 `drizzle-kit generate` + `migrate`。如果没有（当前项目没有），就写建表脚本并执行。
- schema.ts 改完不等于数据库有了，这是个容易忽略的点。

---

## 问题 6：routers.ts 中重复定义路由

**现象：** `server/routers.ts` 中有两段完全相同的 `analytics: router({...})`，第二段覆盖了第一段，不影响运行但造成混乱。

**根因：** 编辑时遗留了重复代码块。

**建议：**

- 提交前过一遍 diff，特别是对长文件做改动后。

---

## 通用的工作流建议

1. **每次改动前先读项目「骨架」文件：**
   - `drizzle.config.ts` → 数据库方案
   - `package.json` → 依赖与脚本
   - `.env` → 环境变量
   - `server/_core/index.ts` → 服务启动流程
   - 这些文件告诉你项目的基本技术选型，一眼就够。

2. **改一个文件后，确认它的消费者（import 了它的模块）不受影响。**
   - 特别是核心模块（db.ts、routers.ts 等）。

3. **用 `npm run dev` 的终端输出做即时验证。**
   - Vite/tsx watch 模块会在你保存文件后立即报告编译错误。

4. **数据库相关的变更一定要落地。**
   - 改 schema → 执行建表/migration
   - 这是 ORM 开发最常见的盲区。

5. **新依赖先确认能否正常加载。**
   - 简单测试：加一行 `import`，看 Vite dev server 有没有报错。

6. **关注现有代码的风格和模式。**
   - 项目已有的函数怎么写、参数怎么传、错误怎么处理——保持一致而不是另起炉灶。
